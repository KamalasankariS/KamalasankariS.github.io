MALCODEAI: AI-POWERED MALICIOUS CODE DETECTION

IEEE IRI · 2025

PROBLEM STATEMENT
  • Malicious code can be semantically disguised within benign-looking programs
  • Traditional static analysis misses intent-level obfuscation
  • No tool combined intent comparison with component-level deep inspection
SOLUTION
  • Two-phase pipeline: intent comparison + program decomposition
  • Phase 1: LLM generates code description, compared with user-stated intent via semantic similarity
  • Phase 2: AST-based decomposition into independent components, each scored and inspected
OUR POSITIONING
  • Novel two-stage pipeline: first decomposes code into functional components, then applies semantic vulnerability detection with red-hat-style exploit reasoning and automatic remediation
  • Language-agnostic approach supporting 14 programming languages
  • Red-hat-style exploit tracing and CVSS-based risk scoring for actionable security assessment
  • Zero-shot generalization capability for detecting complex, zero-day vulnerabilities
  • Positioned against CodeBERT, GraphCodeBERT, GPT-based variants, VulDeePecker, VulBERTa which are constrained to known vulnerabilities or narrow datasets
ARCHITECTURE
  • Phase 1: DeepSeek Coder v2 (1.3B instruct) for code description generation
  • GTR-T5-Large for semantic similarity between generated and user descriptions
  • Phase 2: AST parsing + graph analysis for program decomposition
  • Robustness scoring (1-10) per component; low-scoring components undergo deep inspection
  • Threat classification with explanation and fix recommendation generation
DATASET USED
  • Custom dataset of labeled malicious and benign code snippets across 14 programming languages (Python, Java, C, C++, Rust, Go, Scala, and more)
  • Focused on 10 critical vulnerability types (SQL Injection, XSS, Remote Code Execution, etc.)
  • Remediation evaluation: 20 manually crafted code files spanning 4 mini-projects (Python, C, Java, JavaScript)
  • 11 injected vulnerabilities in the insecure project; 6 in mixed-security setups
  • Qualitative evaluation: feedback from 15 developers (practitioners and CS students)
  • LoRA fine-tuning on Qwen2.5-Coder-3B-Instruct within MLX framework on Apple Silicon M4 Pro
RESULTS
  • Phase 1 (Code Decomposition) best validation loss: 0.397 (200 iterations, 6 trainable layers)
  • Phase 2 (Vulnerability Detection) best validation loss: 0.199 (learning rate 4e-5, 6 trainable layers)
  • Developer survey (n=15): Usefulness 8.06/10, Readability 7.00/10, Interpretability 7.40/10
  • Remediation: 8/11 vulnerabilities detected in insecure project (5 specific fixes, 3 generic); 5/6 in mixed projects
  • System found 1 overlooked vulnerability confirmed upon manual inspection
  • Training time: ~3-4 hours per phase on Apple Silicon M4 Pro
